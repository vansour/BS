# 静止假车误检治理计划（2026-04-27）

## 目标

当前项目的车辆检测主线仍以单帧外观为主，因此会出现一种典型问题：

- 某些长得像车的静止物体会被持续检测为 `vehicle`
- 在视频连续帧中，这类误检会表现为“长时间稳定存在的假车轨迹”

本计划的目标不是承诺“绝对 0 误检”，而是把该问题从“现象级抱怨”收敛成：

- 可定义
- 可测量
- 可压制
- 可回归验证

最终希望达到的状态是：

- 静止假车持续误检显著下降
- 真实静止车辆、缓行车辆和拥堵场景下的车辆召回不被明显破坏
- 统一推理路线、混合推理路线和 benchmark 评估链路使用同一套抑制逻辑

## 问题定义

这里关注的不是普通单帧误检，而是“视频级持续静止误检”。

判定对象：

- 在多帧中被持续检测为 `vehicle`
- 目标位置、尺度变化极小
- 实际上并不是真车

典型来源包括：

- 路边设施
- 反光结构
- 阴影或高亮区域
- 与车辆轮廓相似的背景物体

需要特别强调：

- “静止”本身不能直接判为假车
- 高速路场景中可能存在临停、拥堵、事故停车和缓行车辆
- 因此不能使用“只要不动就删除”的粗糙规则

## 总体策略

治理路径采用四层闭环，而不是只调一个阈值：

1. 先建立静止误检基准，让问题可被稳定评估。
2. 再在推理后处理层增加轨迹级时序验证器，把逐帧检测升级为视频级确认。
3. 对可疑静止轨迹增加二次复核，避免误伤真实静止车辆。
4. 将已发现的静止假车样本回灌到训练侧，形成 hard negative 闭环。

这四层中，真正的第一优先级不是改模型结构，而是先把推理层和评估层补完整。

## 分阶段实施计划

### 第一阶段：建立静止误检 benchmark

目的：

- 先回答“问题有多严重”
- 为后续每一轮改动提供统一对照标准

实施内容：

- 从现有真实雾视频、clear-weather control clips 和 `Route_Eval` 输出中整理三类样本：
  - `静止假车`
  - `真实静止/缓行车辆`
  - `正常运动车辆`
- 在路线评估脚本中新增轨迹级统计，而不再只看每帧检测数
- 新增一套与本问题强相关的指标

建议核心指标：

- `persistent_static_fp_per_min`
  - 每分钟持续静止误检轨迹数
- `fp_dwell_frames`
  - 单个静止误检轨迹持续帧数
- `stopped_vehicle_recall`
  - 真实静止/缓行车辆召回率
- `confirmation_latency`
  - 目标从首次出现到被确认显示的延迟
- `runtime_fps`
  - 增加时序过滤后的视频处理速度

优先接入文件：

- `scripts/evaluate_inference_routes.py`

阶段验收：

- 能输出上述指标
- 能在同一视频、同一权重、同一采样策略下稳定复现
- 能明确区分“检测数下降”和“静止误检真正下降”

### 第二阶段：增加轨迹级时序验证器

目的：

- 解决当前系统“只看单帧外观、不看跨帧行为”的根缺陷

实施内容：

- 新增共享模块：
  - `src/temporal_vehicle_filter.py`
- 对检测结果做跨帧关联，形成轻量级轨迹
- 每条轨迹维护以下状态：
  - `tentative`
  - `confirmed`
  - `suspicious`
  - `suppressed`
- 每条轨迹记录以下特征：
  - 连续命中帧数
  - 连续丢失帧数
  - 平均置信度
  - 中心点位移
  - 框面积变化
  - 局部运动强度
  - 是否位于道路 ROI 内

第一版规则不采用“静止即删除”，而采用：

- 连续命中足够帧数后才进入 `confirmed`
- 长时间近乎不动、局部运动极低、位置和尺度异常稳定、且置信度不高时进入 `suspicious`
- 满足抑制条件后进入 `suppressed`

优先接入文件：

- `src/inference.py`
- `scripts/hybrid_fog_vehicle_infer.py`
- `scripts/evaluate_inference_routes.py`

设计要求：

- 同一套过滤器必须同时服务于：
  - 统一推理路线
  - 混合推理路线
  - 离线 benchmark
- 不能只在 UI 绘制层做“隐藏显示”，否则内部统计与评估仍然会把假车算进去

阶段验收：

- 持续静止误检轨迹数明显下降
- 真实静止车辆不会被大面积误杀
- 引入时序过滤后，统一路线与混合路线的行为一致

### 第三阶段：对可疑静止轨迹做二次复核

目的：

- 解决“长得像车、又几乎不动”的最顽固误检
- 避免把真实停车/拥堵车辆一刀切抑制掉

实施内容：

- 仅对 `suspicious` 轨迹裁剪 patch
- 送入轻量二分类器，判断：
  - `vehicle`
  - `non_vehicle`
- 二分类器只运行在少量可疑轨迹上，而不是全帧全框运行

原因：

- 静止误检问题仅靠运动约束无法彻底解决
- 因为真实静止车辆本来就可能零位移
- 此时需要第二层外观复核，而不是单纯阈值硬删

实现建议：

- 先用轻量 `timm` 分类 backbone 做验证
- 保持与当前主检测模型解耦，避免过早增加统一模型结构复杂度

阶段验收：

- 对“静止像车背景物体”的误检进一步下降
- 对真实停车车辆召回率下降可控
- 整体实时性能仍可接受

### 第四阶段：误检样本回灌与 hard negative 闭环

目的：

- 把后处理层的治理经验沉淀回模型侧
- 避免系统长期依赖规则补丁

实施内容：

- 从第二、第三阶段记录的误检轨迹中自动挖取 hard negatives
- 将静止假车 patch 整理为负样本集
- 新增误检挖掘脚本和结果归档目录
- 形成两个训练方向：
  - 二次分类器 hard negative 增量训练
  - 检测器 / 统一模型的背景抑制增量训练

需要验证的假设：

- 当前单类 `vehicle` 路线是否过度泛化了“像车背景”
- `single_vehicle` 与 `coco_vehicle` 两条路线对静止假车的敏感性是否存在明显差异

阶段验收：

- 对历史已知静止误检样本的压制率提高
- 新样本上的泛化不明显退化
- 模型层改动的收益能在 benchmark 中体现，而不是只在单一视频上有效

## 工程接入顺序

建议按照以下顺序实施，而不是并行大改：

1. 先补 benchmark 指标和轨迹日志
2. 再上线共享时序过滤器
3. 再做可疑轨迹二次复核
4. 最后做 hard negative 回灌

不建议一开始就改 `UnifiedMultiTaskModel` 结构，原因是：

- 当前问题首先是视频时序确认缺失
- 过早改模型会提高训练成本、验证成本和部署复杂度
- 若没有统一 benchmark，模型改动是否有效也无法说清

## 配置项建议

为便于实验对比，建议把以下控制项收口到 `Config`：

- `TEMPORAL_FILTER_ENABLED`
- `TEMPORAL_MIN_HITS`
- `TEMPORAL_MAX_MISSING`
- `TEMPORAL_IOU_MATCH_THRES`
- `TEMPORAL_STATIC_CENTER_SHIFT_THRES`
- `TEMPORAL_STATIC_AREA_CHANGE_THRES`
- `TEMPORAL_STATIC_FRAME_LIMIT`
- `TEMPORAL_LOW_CONF_STATIC_SUPPRESS`
- `TEMPORAL_ENABLE_SECOND_STAGE_CLASSIFIER`

这样可以保证：

- 单模型推理、混合推理、benchmark 可以共用一套配置语义
- 所有结果都能写入 `config_snapshot.json` 和实验摘要

## 风险与约束

主要风险：

- 过强的静止抑制会误伤真实停车车辆
- 轨迹确认机制会引入显示延迟
- 二次复核会增加推理开销
- 不同摄像头视角下，静止阈值和 ROI 规则未必通用

约束原则：

- 任何抑制策略都必须优先保护真实车辆召回
- 任何规则上线前都必须经过 benchmark 对照
- UI 隐藏不等于问题解决，内部统计与输出结果必须同步过滤

## 验收标准

本计划完成后，应至少满足以下条件：

- 静止假车持续误检显著下降
- 真实静止/缓行车辆召回率无明显不可接受退化
- 统一路线、混合路线、benchmark 三边结果一致
- 新增逻辑具备配置化控制能力
- 所有改动都能通过固定 benchmark 复现和回归

## 当前结论

对于本项目，“彻底解决静止假车误检”的正确路径不是：

- 单纯调一个置信度阈值
- 单纯在绘制层隐藏框
- 一开始就重写主模型

而应该是：

- 先让问题可评估
- 再把单帧检测升级成轨迹级确认
- 再对可疑静止轨迹做二次复核
- 最后把误检样本回灌到训练侧

这条路径更慢，但它是当前代码库里最稳、最可证、也最容易沉淀为正式工程能力的方案。
