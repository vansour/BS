\chapter{系统总体设计}

\section{引言}

在明确研究背景、文献现状和理论基础之后，论文需要进一步回答“系统究竟要解决什么问题、整体应如何组织、模块之间如何协同”这类工程层面的核心问题。对本科毕业设计而言，系统总体设计并不是可有可无的过渡内容，而是连接“研究动机”和“代码实现”的关键桥梁。本章将从系统需求分析出发，说明本文为何选择统一模型作为主路线、为何保留混合推理作为工程备份链、以及各功能模块如何围绕训练、推理、导出和误检治理形成一个完整原型系统。

\section{需求分析}

结合高速公路团雾监测的业务需求与当前项目实现条件，本文将系统需求归纳为以下四个层面。

在功能层面，系统需要对视频监控画面中的车辆进行实时检测，同时判断当前天气是否为无雾、均匀雾或局地团雾，并给出一个能够反映雾强度的连续变量。仅输出单一标签无法满足联动管控与视频解释需求，因此检测结果、天气类别与雾浓度必须协同呈现\cite{qx7292024,qx8082025,sunxiao2024}。

在数据层面，系统需要在不具备大规模真实团雾样本的前提下完成训练，这要求数据构建方式具备可复用、可扩展、可持续的特点。换言之，不能将系统建立在“必须不断人工采集真实团雾视频并逐帧标注”的高成本前提上。

在算法层面，系统需要尽量减少模型数量与部署链路复杂度。若车辆检测、天气识别和雾浓度估计分别由三个独立模型承担，则推理链会更长、维护成本更高，也不利于部署和答辩展示。因此，统一模型是更合理的总体方向。

在工程层面，系统不仅要能训练，还必须能推理、能导出、能可视化。毕业设计不仅关注模型思想是否成立，还关注完整工程链路是否打通。为此，数据审计、深度预计算、训练脚本、推理脚本、混合推理、视频导出、QAT 与 ONNX 导出都应当纳入总体设计。

\section{技术路线}

本文系统总体技术路线可概括为：

\begin{enumerate}[leftmargin=2.5em]
\item 从 UA-DETRAC 中读取晴天交通监控图像与 XML 标注\cite{uadetrac}；
\item 使用 \code{MiDaS_small} 对全部样本离线估计深度，并保存深度缓存\cite{midas}；
\item 在训练阶段利用深度图和大气散射模型执行在线造雾；
\item 将雾化图像输入统一多任务模型，联合优化检测、雾分类和 beta 回归；
\item 在视频推理阶段，对输入帧执行联合前向，输出天气类别、beta 和车辆检测框；
\item 在后处理阶段对检测框执行 NMS、轨迹级时序过滤和可选二次外观复核；
\item 对 beta 进行时间平滑，并基于其调整显示阈值；
\item 将结果视频输出为可视化演示文件，并保留导出到 ONNX、QAT/INT8 的接口；
\item 对真实视频中被抑制的静止可疑轨迹进行 patch 挖掘、人工复核与 hard negative 回灌。
\end{enumerate}

这一技术路线兼顾了物理启发、深度学习表示能力和工程部署要求，避免了完全依赖手工特征或完全依赖难获取真实雾数据的极端方案。

\section{模块划分}

从代码实现角度看，本文系统可划分为以下几个核心模块。

\begin{enumerate}[leftmargin=2.5em]
\item 配置模块。位于 \code{src/config.py}，统一管理数据路径、模型参数、训练超参数、物理参数、推理阈值和导出目录，是训练、推理和导出的统一真实配置源。
\item 数据模块。位于 \code{src/data/}。其中 \code{dataset.py} 负责构建多任务训练数据集，\code{depth_estimator.py} 负责深度估计与缓存生成，\code{preparer.py} 则提供离线 YOLO 数据集整理功能。
\item 模型模块。位于 \code{src/model/}。其中 \code{fog_augmentation.py} 负责在线造雾增强，\code{unified_model.py} 负责统一多任务模型结构定义。
\item 训练模块。位于 \code{src/train.py}，负责数据装载、损失计算、联合训练、续训、QAT 和权重保存。
\item 推理模块。位于 \code{src/inference.py}，负责加载统一模型并对视频帧进行推理与可视化。
\item 后处理与误检治理模块。位于 \code{src/temporal_vehicle_filter.py}，用于对车辆检测结果做轨迹级时序过滤、静止候选判别和二次外观复核。
\item 辅助脚本模块。位于 \code{scripts/}，用于实现数据审计、深度缓存补算、训练前自检、混合推理、路线评估、静止误检 patch 挖掘、人工复核包构建以及二次复核分类器训练等。
\end{enumerate}

\section{系统特点}

本文系统具有以下几个鲜明特点。

其一，真实配置源统一。与许多“文档配置和代码配置并存、易造成版本漂移”的项目不同，本文明确以 \code{src/config.py} 作为真实配置入口，使训练、推理和导出均围绕统一参数展开。

其二，训练与推理几何空间保持一致。无论是数据集侧的 letterbox 处理，还是推理时的视频帧预处理，本文均采用一致的几何对齐方式，从而降低训练-推理几何失配带来的性能波动。

其三，系统从一开始就考虑了部署和容错。例如，训练脚本支持缺失深度缓存自动补算；推理系统支持权重自动回退；导出脚本支持 ONNX 输出；量化路径保留了 QAT 与 INT8 转换能力。这使得本项目不仅是一个研究方法，也是一套具备工程闭环特征的原型系统。

其四，系统已初步具备视频级误检治理能力。当前版本不再停留在“每帧独立检测”的层面，而是通过轨迹级时序过滤和 patch 级二次复核，对真实监控视频中的静止假车误检进行抑制，并通过日志、patch 和 hard negative 数据回灌形成闭环。

其五，系统具有较强的扩展性。后续若要引入真实雾视频监督、加入可见度等级估计、融合更多交通气象传感器数据，或将二次复核器替换为更大规模的专用分类器，均可在当前框架上继续演进，而无需完全推翻已有实现。

\section{应用场景与部署需求}

若将本文系统放到陕西交通运输部门的现实业务框架下，其价值并不只是“做出一个能演示的雾识别模型”，而在于能否作为现有路网运行监测体系、收费与出行服务体系、气象预警转发体系和“一路多方”协同体系中的一个新增能力模块。根据陕西省交通运输厅 2026 年 1 月发布的工作会议信息，全省高速公路通车里程已达 6895 公里；根据 2025 年末路网周报，西安绕城、新机场线、临潼—兴平、连霍高速和西安外环等线路长期维持高交通量状态；根据 2025 年高速公路监测预警系统建设（第一批）招标公告，全省已开始面向 18 家路段分公司的 142 处风险点建设监测预警系统\cite{shaanxiMeeting2026,shaanxiWeekly2025,shaanxiRiskProject2025}。上述现实条件共同说明：陕西高速公路管理已进入“基础设施规模持续扩大、视频资源持续增加、监测预警系统持续建设”的阶段，具备引入本文项目的现实基础。

从实际部署顺序看，陕西场景更适合采用“先视频增量、后系统融合”的渐进式策略，而不是一次性替换既有监测体系。第一阶段可优先在交通量大、监控条件成熟的西安绕城、新机场线、连霍高速关中段等线路开展验证，重点考察视频流接入、推理稳定性、结果解释性和指挥调度可接受性；第二阶段可向桥隧比高、天气变化快、跨省联动需求强的陕南山区通道拓展，特别是岚皋—陕渝界、大巴山隧道周边及安康、汉中、商洛等区域的典型监控点位；第三阶段再考虑与省级路网运行中心、收费中心或交控集团监测预警工程做更深层系统集成。

从系统接口角度看，本文项目也较符合陕西现有信息化体系的接入逻辑。当前陕西交通系统已具备路网周报、监测预警建设、出行服务平台和应急转发机制，因此本文项目完全可以定位为“算法侧插件”而不是“平台侧重构工程”：输入端接入既有 CCTV 或监控视频流，算法端输出雾类型、beta、检测结果和风险级别，平台端再决定是否触发限速诱导、情报板提示、值班提醒或会商研判。这种松耦合方式既能降低落地门槛，也更符合交通部门在现有系统上逐步叠加新能力的现实路径。

最后，需要强调的是，陕西交通部门当前推进的监测预警工程更多偏向结构安全、灾害点监测和综合感知，而本文项目聚焦于“低能见度视频智能感知”。两者并非竞争替代关系，而是互补关系：前者更关注基础设施与路网安全状态，后者更关注固定视频中车辆可见性、天气状态和通行风险的实时表达。若未来在陕西场景中开展进一步应用验证，本文项目最合理的定位应当是既有监测预警体系中的“视频雾风险感知模块”。

\section{本章小结}

本章从系统需求、技术路线、模块划分、系统特点以及陕西交通部门应用场景五个层面对本文原型系统进行了总体设计说明。通过本章分析可以看出，本文并不是单纯提出一个雾识别模型，而是围绕高速公路固定监控视频构建了一条从数据准备、模型训练到视频推理、导出部署和误检治理的完整系统链路。该总体设计为后续数据集构建、在线团雾合成、统一多任务模型设计以及实验验证奠定了工程框架。
